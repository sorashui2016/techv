# 科技选题雷达项目上下文

更新时间：2026-05-18

本文档用于在聊天记录丢失或上下文压缩后恢复项目方向。它基于用户提供的《科技选题雷达系统产品目标 SPEC v3.0》，但不是最终版。后续讨论产生的新需求、取舍和实现细节都应继续追加到这里。

## 当前定位

项目名称：科技选题雷达

目标用户：科技/数码内容创作者。

核心目标：定时监测用户关注的科技类账号、频道和信息源，自动抓取最新视频内容，翻译标题，生成中文摘要，进行初步评分，并在网站后台以卡片形式呈现，帮助用户快速判断哪些内容值得作为选题。

第一阶段重点不是完整 AI 研究系统，而是先做好稳定的监测后台：

```text
配置监测账号
→ 按账号类型定时监测
→ 抓取新视频
→ 翻译标题
→ 生成 100 字内中文总结
→ 初步评分
→ Dashboard 卡片展示
→ 用户标记状态
→ 进入不同状态池
→ 预留研究入口和飞书接口
```

## 三阶段路径

第一阶段：科技选题雷达

- 定时监测账号和频道。
- 抓取视频内容。
- 翻译标题。
- 生成中文摘要。
- 初步评分。
- 卡片式展示。
- 状态池管理。
- 预留飞书推送接口。
- 预留研究入口。

第二阶段：选题深度研究

- 用户点击“研究”后，系统围绕该选题进行全网资料搜索、整理、事实核查和研究报告生成。
- 当前只预留入口，不细化完整实现。

第三阶段：素材与内容创作

- 在研究基础上继续完成素材搜索、授权标注、字幕/配音翻译、视频文案生成等内容生产工作。
- 当前不作为第一阶段范围。

## 第一阶段用户能力

第一阶段完成后，用户应该可以：

- 管理需要监测的账号。
- 区分普通账号和重要账号。
- 自动定时监测新视频。
- 在 Dashboard 上按监测时间查看最新内容。
- 看到视频封面、原始标题、中文标题、中文摘要、发布时间、点赞数、初步评分、来源平台、来源账号、监测时间等信息。
- 点击卡片或明显入口打开原始链接。
- 给每条内容标记状态：备选、已做、待定、不做。
- 在不同状态池中查看内容。
- 点击“研究”按钮，为第二阶段研究功能预留入口。
- 手动提交视频链接进入雷达池。
- 查看基础监测日志和错误信息。

## 账号分类与监测频率

普通账号：

- 北京时间每天 08:00 监测一次。
- 有更新时只进入网站后台。
- 第一阶段不主动推送飞书。

重要账号：

- 北京时间 00:00 / 04:00 / 08:00 / 12:00 / 16:00 / 20:00 监测。
- 有更新时第一阶段可先记录为“待推送”。
- 第一阶段不强制做飞书推送，但代码结构要为后续推送做准备。

## 平台支持范围

优先支持：

- YouTube
- RSS 视频源
- 普通网页视频源或科技媒体视频页
- Product Hunt / Kickstarter / Indiegogo 等产品页的视频内容，如果技术上容易实现可纳入

有限支持：

- Instagram：只关注带视频内容；图文内容不进入推送逻辑；不要求 Stories、私密内容、登录受限内容。
- Twitter/X：可作为有限支持；优先关注带视频推文或科技账号更新；如访问不稳定，允许降级为仅保存链接或人工提交。

第一阶段暂不强制完整支持：

- B站
- 小红书
- 抖音
- TikTok
- Facebook
- Reddit
- Hacker News

注意：当前代码已包含 TikTok 枚举和手动链接支持，但产品上它不是第一阶段硬性完整支持平台。

## 视频入库字段

每条视频至少应包含：

- 原始链接
- 视频封面图片
- 原始标题
- 中文标题翻译
- 视频介绍/描述
- 视频内容中文总结
- 发布时间
- 点赞数
- 来源平台
- 来源账号
- 监测时间
- 初步评分
- 当前状态

标题翻译要求：

- 抓取后立即生成并存储。
- 页面展示时不临时翻译。
- 后台卡片同时展示原始标题和中文标题。
- 翻译失败时保留原始标题，并记录处理状态，允许后续重试。

中文总结要求：

- 根据标题、描述、字幕、平台元数据等生成。
- 中文输出。
- 不超过 100 字。
- 说明视频讲了什么，避免空泛营销话术。
- 优先提炼产品、技术、观点、亮点或争议点。

评分要求：

- 使用 0-100 分。
- 目标是辅助筛选，不追求绝对准确。
- 可综合来源重要度、新鲜度、标题信号、描述信息密度、点赞互动、科技选题价值、视觉素材潜力、争议点、趋势价值、用户关注方向等。

去重要求：

- 避免重复展示同一视频。
- 可依据原始链接、平台视频 ID、标题相似度、来源账号 + 发布时间。
- 重复监测到同一内容时，更新监测时间或互动数据，不重复生成卡片。

## Dashboard 要求

默认排序：

- 按监测时间倒序。

后续可扩展排序：

- 发布时间
- 评分
- 点赞数
- 来源重要度

卡片展示：

- 视频封面图片
- 原始标题
- 中文标题
- 100 字内中文总结
- 发布时间
- 点赞数
- 初步评分
- 来源平台
- 来源账号
- 监测时间

卡片行为：

- 点击卡片主体应打开原始链接，或在详情页中提供明显原始链接入口。
- 第一阶段建议尽量降低操作成本。

状态按钮：

- 备选
- 已做
- 待定
- 不做

状态池：

- 全部
- 备选
- 已做
- 待定
- 不做
- 未标记

建议筛选能力：

- 按状态筛选
- 按平台筛选
- 按账号筛选
- 按评分筛选
- 按时间筛选
- 关键词搜索

研究按钮：

- 每张卡片有“研究”按钮。
- 第一阶段可以创建研究任务占位记录并进入占位页。
- 目的是为第二阶段研究功能预留入口。

## 后台管理要求

信息源管理字段：

- 名称
- 平台
- 账号链接或频道链接
- 账号类型：普通账号 / 重要账号
- 是否启用
- 最近监测时间
- 最近监测状态
- 备注

信息源操作：

- 新增
- 编辑
- 禁用
- 删除
- 修改账号类型
- 手动触发一次监测

新增信息源体验：

- 来源名称可以手动填写。
- 如果用户不填写来源名称，系统应自动从账号/频道链接识别名称。
- YouTube 优先读取 yt-dlp 返回的频道显示名，例如 `https://www.youtube.com/@tesla` 应识别为 `Tesla`；识别失败时再用链接中的 handle 或域名兜底。
- 用户可以一次粘贴多个 YouTube 频道/账号链接，支持换行、空格或逗号分隔；系统逐个识别名称并入库。
- 批量新增时，重复链接或已存在的信息源应跳过，不应导致整个批次失败。

监测日志：

- 最近一次监测时间
- 是否成功
- 发现了多少新视频
- 更新了多少旧视频
- 错误原因简述

手动提交链接：

- 用户提交视频链接后，系统尽量执行与自动抓取相同的处理：抓取基础信息、翻译标题、生成中文总结、评分、展示为卡片。

## 飞书预留

第一阶段先不做实际飞书推送。

但架构要预留：

- 重要账号有新视频时推送飞书。
- 每日雷达摘要推送。
- 飞书卡片操作：备选、已做、待定、不做、研究。
- 用户通过飞书提交链接。

当前规则预留：

- 重要账号新视频标记为待推送。
- 普通账号新视频只进入后台。

## 当前代码现状

已实现或已初步实现：

- Next.js App Router 项目。
- Prisma/PostgreSQL schema。
- Dashboard 首页。
- 信息源管理页。
- 监测日志页。
- 研究任务占位页。
- 手动提交链接表单。
- 视频状态按钮：备选、已做、待定、不做。
- 已看状态：点击原始链接会标记 VIEWED。
- 来源管理：新增、禁用/启用、删除、手动监测。
- 监测逻辑：普通账号 08:00；重要账号 00/04/08/12/16/20。
- yt-dlp 抓取视频信息。
- Minimax 生成中文标题、摘要、评分；未配置 API key 时使用 fallback。
- 重要账号视频 `pushStatus` 预留为 `PENDING`。
- 研究按钮创建 `ResearchTask` 并跳转占位页。

已验证：

- `npm.cmd run lint` 通过。
- `npm.cmd run build` 通过。
- 本地 `http://localhost:3000` 可访问。
- `/sources` 和 `/logs` 可访问。

明显待补或待确认：

- 信息源暂时只有新增、禁用/启用、删除，没有完整编辑表单。
- Dashboard 当前有状态和平台筛选、关键词搜索，但还没有账号、评分、时间筛选。
- 当前原始链接是按钮打开，不是整张卡片主体点击。
- 当前平台枚举包含 TikTok，但文档中 TikTok 不作为完整支持硬需求。
- RSS、普通网页、Product Hunt / Kickstarter / Indiegogo 抓取能力尚未确认。
- Twitter/X 尚未纳入 schema 枚举，需要后续决定是否加入第一阶段有限支持。
- 字幕获取和基于字幕总结尚未确认。
- 飞书只有数据结构预留，没有独立接口模块。
- 生产部署、域名访问、定时任务部署方式还未定。
