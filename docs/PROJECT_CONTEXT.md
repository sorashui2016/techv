# 科技选题雷达项目上下文

更新时间：2026-05-19

本文档用于在聊天记录丢失或上下文压缩后恢复项目方向。它基于用户提供的《科技选题雷达系统产品目标 SPEC v3.0》，但不是最终版。后续讨论产生的新需求、取舍和实现细节都应继续追加到这里。

## 当前定位

项目名称：科技选题雷达

目标用户：科技/数码内容创作者。

核心目标：定时监测用户关注的科技类账号、频道和信息源，自动抓取最新视频内容，翻译标题，生成中文摘要，进行初步评分，并在网站后台以卡片形式呈现，帮助用户快速判断哪些内容值得作为选题。

第一阶段重点不是完整 AI 研究系统，而是先做好稳定的监测后台：

```text
配置监测账号
→ 按账号类型定时监测
→ 抓取新视频
→ 翻译标题
→ 生成 100 字内中文总结
→ 初步评分
→ Dashboard 卡片展示
→ 用户标记状态
→ 进入不同状态池
→ 预留研究入口和飞书接口
```

## 三阶段路径

第一阶段：科技选题雷达

- 定时监测账号和频道。
- 抓取视频内容。
- 翻译标题。
- 生成中文摘要。
- 初步评分。
- 卡片式展示。
- 状态池管理。
- 预留飞书推送接口。
- 预留研究入口。

第二阶段：选题深度研究

- 用户点击“研究”后，系统围绕该选题进行全网资料搜索、整理、事实核查和研究报告生成。
- 当前只预留入口，不细化完整实现。

第三阶段：素材与内容创作

- 在研究基础上继续完成素材搜索、授权标注、字幕/配音翻译、视频文案生成等内容生产工作。
- 当前不作为第一阶段范围。

## 第一阶段用户能力

第一阶段完成后，用户应该可以：

- 管理需要监测的账号。
- 区分普通账号和重要账号。
- 自动定时监测新视频。
- 在 Dashboard 上按监测时间查看最新内容。
- 看到视频封面、原始标题、中文标题、中文摘要、发布时间、点赞数、初步评分、来源平台、来源账号、监测时间等信息。
- 点击卡片或明显入口打开原始链接。
- 给每条内容标记状态：备选、已做、待定、不做。
- 在不同状态池中查看内容。
- 点击“研究”按钮，为第二阶段研究功能预留入口。
- 手动提交视频链接进入雷达池。
- 查看基础监测日志和错误信息。

## 账号分类与监测频率

普通账号：

- 北京时间每天 08:00 监测一次。
- 有更新时只进入网站后台。
- 第一阶段不主动推送飞书。

重要账号：

- 从北京时间 08:00 开始，每 6 小时监测一次：08:00 / 14:00 / 20:00 / 02:00。
- 有更新时第一阶段可先记录为“待推送”。
- 第一阶段不强制做飞书推送，但代码结构要为后续推送做准备。

## 平台支持范围

优先支持：

- YouTube
- RSS 视频源
- 普通网页视频源或科技媒体视频页
- Product Hunt / Kickstarter / Indiegogo 等产品页的视频内容，如果技术上容易实现可纳入

有限支持：

- Instagram：只关注带视频内容；图文内容不进入雷达池或推送逻辑；不要求 Stories、私密内容、登录受限内容。当前本机 yt-dlp 显示 `instagram:user (CURRENTLY BROKEN)`，因此账号页自动监测不能视为稳定支持；单条公开视频/Reels 链接可尝试处理。
- Twitter/X：可作为有限支持；优先关注带视频推文或科技账号更新；如访问不稳定，允许降级为仅保存链接或人工提交。

第一阶段暂不强制完整支持：

- B站
- 小红书
- 抖音
- TikTok
- Facebook
- Reddit
- Hacker News

注意：当前代码已包含 TikTok 枚举和手动链接支持，但产品上它不是第一阶段硬性完整支持平台。

## 视频入库字段

每条视频至少应包含：

- 原始链接
- 视频封面图片
- 原始标题
- 中文标题翻译
- 视频介绍/描述
- 视频内容中文总结
- 发布时间
- 点赞数
- 来源平台
- 来源账号
- 监测时间
- 初步评分
- 当前状态

标题翻译要求：

- 抓取后立即生成并存储。
- 页面展示时不临时翻译。
- 后台卡片同时展示原始标题和中文标题。
- 翻译失败时保留原始标题，并记录处理状态，允许后续重试。

中文总结要求：

- 根据标题、描述、字幕、平台元数据等生成。
- 中文输出。
- 不超过 100 字。
- 说明视频讲了什么，避免空泛营销话术。
- 优先提炼产品、技术、观点、亮点或争议点。

评分要求：

- 使用 0-100 分。
- 目标是辅助筛选，不追求绝对准确。
- 可综合来源重要度、新鲜度、标题信号、描述信息密度、点赞互动、科技选题价值、视觉素材潜力、争议点、趋势价值、用户关注方向等。

去重要求：

- 避免重复展示同一视频。
- 可依据原始链接、平台视频 ID、标题相似度、来源账号 + 发布时间。
- 重复监测到同一内容时，更新监测时间或互动数据，不重复生成卡片。

## Dashboard 要求

默认排序：

- 按监测时间倒序。

后续可扩展排序：

- 发布时间
- 评分
- 点赞数
- 来源重要度

卡片展示：

- 视频封面图片
- 原始标题
- 中文标题
- 100 字内中文总结
- 发布时间
- 点赞数
- 初步评分
- 来源平台
- 来源账号
- 监测时间

卡片排序和视觉：

- Dashboard 中重要账号的视频应排在普通账号视频前面。
- 重要账号视频卡片使用浅绿色背景。
- 已看过但仍未标记的视频优先使用更深的灰色背景，文字变灰，和其他未处理卡片明显区分。

卡片行为：

- 卡片上不需要单独的“原始链接”按钮。
- 点击封面图片、中文标题、原始标题或中文摘要，都应标记为已看并在新标签页打开原始链接。
- 状态按钮和研究按钮保持独立操作，不应触发打开原始链接。

状态按钮：

- 备选
- 已做
- 待定
- 不做

状态池：

- 全部
- 备选
- 已做
- 待定
- 不做
- 未标记

首页默认行为：

- Dashboard 首页默认只显示“未标记”的待处理内容。
- 用户点击卡片上的“备选 / 已做 / 待定 / 不做”后，该内容应进入对应状态池，并从首页默认视图消失。
- 用户可以通过状态筛选查看“全部状态”或单独的备选、已做、待定、不做池子。

建议筛选能力：

- 按状态筛选
- 按平台筛选
- 按账号筛选
- 按评分筛选
- 按时间筛选
- 关键词搜索

研究按钮：

- 每张卡片有“研究”按钮。
- 第一阶段可以创建研究任务占位记录并进入占位页。
- 目的是为第二阶段研究功能预留入口。

## 后台管理要求

信息源管理字段：

- 名称
- 平台
- 账号链接或频道链接
- 账号类型：普通账号 / 重要账号
- 是否启用
- 最近监测时间
- 最近监测状态
- 备注

信息源操作：

- 新增
- 编辑
- 禁用
- 删除
- 修改账号类型，普通账号和重要账号可以随时互相切换
- 手动触发一次监测
- 支持“全部手动监测”：对所有启用信息源各抓取一次最新视频；已抓取过的视频不重复显示，只更新已有记录。

新增信息源体验：

- 来源名称可以手动填写。
- 如果用户不填写来源名称，系统应自动从账号/频道链接识别名称。
- 新增信息源表单应尽量紧凑，不要占用太高空间；批量链接输入可以是单行输入框，但仍需支持粘贴多个链接。
- YouTube 优先读取 yt-dlp 返回的频道显示名，例如 `https://www.youtube.com/@tesla` 应识别为 `Tesla`；识别失败时再用链接中的 handle 或域名兜底。
- 用户可以一次粘贴多个 YouTube 频道/账号链接，支持换行、空格或逗号分隔；系统逐个识别名称并入库。
- 批量新增时，重复链接或已存在的信息源应跳过，不应导致整个批次失败。
- YouTube 去重不能只看原始 URL；同一频道可能以 `@handle`、`/channel/UC...`、`/user/...` 等形式提交。系统应通过 yt-dlp 解析真实 channel id，保存为 `sourceKey`，并用它防止重复添加。
- 无效 YouTube 频道链接不应入库；如果 yt-dlp 返回频道不存在或无法识别名称，批量结果中应标记为 failed，并保留错误原因。
- 如果已存在的信息源监测失败，信息源列表应醒目标记“需要处理”，提示用户删除或替换为正确链接。
- 信息源应提供“替换链接”操作；替换时必须重新校验并识别来源名，坏链接不能保存。
- 信息源列表不直接显示账号链接，也不展示失败原因详情；失败项只显示“failed · 需要处理”。
- 点击信息源名称应在新标签页打开账号页面。
- 信息源列表中重要账号显示在普通账号前面，并使用浅灰底色区分。
- “替换链接”按钮只在坏链接或监测失败的信息源上显示。

监测日志：

- 最近一次监测时间
- 是否成功
- 发现了多少新视频
- 更新了多少旧视频
- 错误原因简述

本地定时任务：

- 当前 Windows 本机应设置 `TechV Daily Monitor` 计划任务，从每天本机时间 08:00 开始每 6 小时执行 `npm.cmd run monitor`。
- 该任务调用 `monitorDueSources()`；北京时间 08:00 时普通账号和重要账号都会被监测，14:00 / 20:00 / 02:00 只监测重要账号。
- 任务日志输出到 `logs/monitor-task.log`，`logs/` 不进入 Git。
- 这是第一阶段本机方案；账号增多或部署上线后，应升级为任务队列/worker 模式，而不是让一个 cron 串行跑完所有账号。

手动提交链接：

- 用户提交视频链接后，系统尽量执行与自动抓取相同的处理：抓取基础信息、翻译标题、生成中文总结、评分、展示为卡片。

## 飞书推送

第一阶段支持可选的飞书自定义机器人 Webhook 推送，但不做完整飞书交互应用。

后续仍需预留：

- 重要账号有新视频时推送飞书。
- 每日雷达摘要推送。
- 飞书卡片操作：备选、已做、待定、不做、研究。
- 用户通过飞书提交链接。

当前规则：

- 重要账号新视频在未配置 Webhook 时标记为待推送。
- 普通账号新视频只进入后台。

当前实现：

- 支持飞书自定义机器人 Webhook。
- 配置 `FEISHU_WEBHOOK_URL` 后，重要账号“新入库”的视频会自动推送到飞书；重复抓取到已存在视频不会重复推送。
- 如果飞书机器人启用了签名校验，配置 `FEISHU_WEBHOOK_SECRET`。
- 推送成功或失败都会写入 `PushEvent`；成功后视频 `pushStatus` 为 `SENT`，失败为 `FAILED`。
- 未配置 Webhook 时，重要账号视频保持 `PENDING`，不实际推送。

## 平台探索雷达 / 科技发现流

本模块是在“指定账号监测”之外新增的探索型雷达功能。账号监测负责不漏掉用户已关注的信息源；平台探索雷达负责主动在平台上搜索用户尚未关注但可能值得研究的视频和科技内容。

已确认第一版范围：

- 第一版先只做 YouTube 搜索，RSS / Product Hunt / Kickstarter / Indiegogo / TED 等平台先预留扩展接口。
- 搜索规则、加分规则、降权规则、排除规则、权威来源规则都必须存入数据库并可在后台编辑，不能写死在代码里。
- 每个搜索关键词第一版抓取 10 条；每次探索最多处理 100 条候选。
- 可以预置一批 30-50 条初始探索规则，但这些规则也必须入库，用户可编辑、禁用或删除。
- 探索内容不进入现有首页雷达，单独放在 `/explore` 相关页面，避免和已关注账号监测内容混在一起。
- 探索内容的“备选 / 待定 / 不做”等状态第一版先独立于首页视频状态池，后续再考虑统一选题池。
- 今日推荐 10 条每天生成后固定，不随页面刷新随机变化；手动重新探索时可以重新生成。
- 第一版探索雷达暂时不推送飞书，只在网站内展示；推荐质量稳定后再做“每日探索 10 条”飞书推送。
- 定时探索建议每天北京时间 08:30 执行，和 08:00 账号监测错开。

建议新增页面：

- `/explore`：探索雷达首页，显示今日推荐 10 条、最近候选、高权威低热度、高评分内容。
- `/explore/next`：下一条探索模式，每次只展示一条候选内容。
- `/explore/rules`：探索规则管理，支持新增、编辑、禁用、删除规则，以及手动触发探索搜索。

建议新增数据模型：

- `ExploreRule`：探索规则。字段包括 keyword、type、category、weight、platform、status、notes。
- `ExploreCandidate`：探索候选。字段包括 platform、platformContentId、originalUrl、thumbnailUrl、originalTitle、chineseTitle、description、chineseSummary、publishedAt、likeCount、viewCount、sourceName、sourceType、score、scoreReason、tags、matchedRules、recommendationReason、status、isTodayPick、discoveredAt、lastSeenAt。
- `ExploreRun`：探索运行日志。字段包括 startedAt、finishedAt、status、searchedRuleCount、candidateCount、newCandidateCount、errorMessage。
- 后续可加 `ExploreAction` 记录用户行为：研究、备选、待定、不做、下一条、打开原始链接。

第一版工作流：

```text
用户配置探索关键词和规则
→ 每天 08:30 或手动触发 YouTube 搜索
→ 抓取候选视频基础信息
→ 去重和过滤
→ 翻译标题
→ 生成 100 字以内中文摘要
→ 生成标签、推荐理由和初步评分
→ 写入探索候选池
→ 生成今日推荐 10 条
→ 用户在 `/explore` 或 `/explore/next` 判断：研究 / 备选 / 待定 / 不做 / 下一条 / 打开原始链接
```

第一版推荐策略：

- 不只按点赞数或观看数排序。
- 同时考虑热度、来源权威性、研究价值、视觉表达潜力和用户选题偏好。
- 今日推荐建议混合：3 条高热度科技内容、3 条高权威低热度内容、2 条众筹/新产品内容、2 条 DIY/实验室内容；某类不足时用其他高分内容补足。
- “下一条探索”采用半随机策略：70% 高评分内容，20% 高权威低热度内容，10% 随机探索内容。

第一版重点方向：

- 科技新闻与前沿技术：便携式 MRI、医疗机器人、脑机接口、AI 医疗、能源科技、材料科技。
- 权威新闻与大学研究：Yale Medicine、CBC News、MIT、Stanford、Mayo Clinic、Johns Hopkins、Harvard Medical School 等。
- 众筹科技产品：AI 宠物、智能硬件、桌面机器人、新奇可穿戴设备。
- DIY / Maker / 极客项目：自制机器人、机械臂、Arduino / Raspberry Pi 项目、工程师自制设备。
- 科学技术探秘：Neuralink、神经科学实验室、医疗机器人实验室、科研设备展示。
- TED / TEDx 科技演讲：医疗科技、机器人、脑机接口、AI 与人类未来。

和账号监测模块的关系：

- 本功能不替代账号监测，两者并行存在。
- 如果探索雷达发现的视频来自已监测账号，应尽量复用或关联已有账号信息，避免重复。
- 后续可在探索候选卡片上增加“加入监测账号”按钮，第一版可先不做。

- 信息源管理页。
